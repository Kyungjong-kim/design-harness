# 디자인시스템 문서 트리 템플릿

> 라이브러리/디자인시스템의 **캐노니컬 문서 트리** 골격. `agent/design-system.md`(에이전트 quickref)가 커지면 이 트리로 분리해 "원칙·토큰·컴포넌트 사양·조합 패턴"을 SoT 단위로 둔다.
> 위치: `docs/design-system/` (단일 패키지) 또는 `<package>/docs/design-system/`.

---

## 언제 도입하나

| 신호 | 조치 |
|---|---|
| `agent/design-system.md` 단일 파일이 토큰·컴포넌트·패턴까지 다 떠안아 비대 | 이 트리로 분리. agent 문서는 quickref로 축소 |
| 컴포넌트 20개 이상 — Storybook autodocs만으로 "언제 쓰나/조합 규칙" 부족 | `components/`·`patterns/` 신설 |
| 외부 사용자(채용·동료)가 라이브러리 신뢰도를 문서로 판단 | foundation/tokens 정면 문서로 첫인상 강화 |

---

## 디렉토리 골격

```
docs/design-system/
├── README.md            # 개요·구조·네이밍 컨벤션·SoT 규칙
├── foundation/          # "왜·어떻게" 원칙
│   ├── principles.md    # 토큰만 사용·DS 재사용·disabled 패턴·AI 클리셰 회피·안티패턴 표
│   ├── color.md         # 팔레트(core) + 의미 색(semantic) 매핑
│   ├── typography.md    # 타이포 스케일·weight 규칙
│   ├── layout-grid.md   # 간격·정렬 원칙
│   ├── motion.md        # 전환·prefers-reduced-motion (토큰 있으면 반영)
│   └── a11y.md          # 자동 보장(Radix 등) vs 호출부 책임 경계, 키보드·focus-visible
├── tokens/              # "값" SoT 가이드
│   ├── _convention.md   # 2계층(core→semantic) 원칙·네이밍·신규 토큰 추가 절차
│   ├── core.md          # raw 팔레트·치수·타이포 토큰 (실측 반영)
│   └── semantic.md      # 의미 토큰 (text/bg/border/interactive 등, 실측 반영)
├── components/          # "부품" 사양 (컴포넌트별)
│   ├── README.md        # 인덱스·카테고리 표
│   └── <name>.md        # import·Props 표·예제·variant 매트릭스·when-to-use·a11y 노트
└── patterns/            # "조합 규칙"
    ├── README.md        # 패턴 인덱스·빠른 선택
    ├── form-layout.md · list-layout.md · detail-layout.md
    ├── empty-state.md · confirm-modal.md · error-handling.md
    └── …
```

> `_` 접두사 = 메타 문서(규칙 자체). 디렉토리는 단수 영역명. 파일명 kebab-case.

---

## SoT 규칙 (직접 수정 위치)

| 영역 | SoT | 비고 |
|---|---|---|
| 토큰 값 | `src/styles/tokens/*.css` | 문서는 설명/링크만 (값 중복 금지) |
| 컴포넌트 Props | 소스 + Storybook autodocs | 문서는 사양·예제·when-to-use |
| 원칙·패턴 | 이 트리 | — |

---

## 작성 원칙 (FILL 시 주의)

1. **토큰 예시는 실제 토큰명만.** 문서 작성 전 `tokens/*.css`를 읽고 실존 토큰만 인용 — 날조 금지(없으면 "추가 후보"로 명시).
2. **컴포넌트 spec은 실제 Props에서 추출.** cva variants·interface를 소스에서 확인.
3. **패턴은 보유 컴포넌트 조합으로.** 미보유 부품에 의존하는 패턴은 일반 요소로 강등하거나 제외.
4. **하네스 배선.** `CLAUDE.md` 진입 문서맵 + 갱신 트리거에 이 트리를 등록 — "새 컴포넌트 추가 시 `components/<name>.md` 기재".
5. **agent quickref와 분리.** `agent/design-system.md`는 빠른 규칙(토큰·variant·절차), 상세 사양·원칙은 이 트리. 상호 링크.

## 마스킹 (공개 자산 적용 시)

라이브러리가 사내 DS에서 파생됐다면 문서 이식 시 회사·제품·도메인 식별자 0건 검증:

```bash
grep -rilE "<회사명>|<제품명>|<사내 서비스>" docs/design-system && echo "마스킹 위반"
```

사내 전용 토큰(브랜드 로고·인프라 색 등)·사업 도메인 예시(특정 업무 흐름)는 중립 예시로 치환한다.
