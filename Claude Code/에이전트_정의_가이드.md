# 커스텀 에이전트 정의 가이드 — 라이브러리 전용

`~/.claude/agents/` (전역) 또는 `<project-root>/.claude/agents/` (프로젝트) 폴더에 위치하는 커스텀 에이전트 정의 방법.

> **design-harness 적용 안내**
> 본 가이드는 본체 하네스의 가이드를 라이브러리 전용으로 보강한 것이다.
> - 유형 1~3 — FE/BE 앱 또는 모노레포 안 라이브러리 패키지에서 활용 (보존)
> - **유형 4·5 — 라이브러리 단독 케이스에 직접 적용** (디자인시스템·UI 키트·SDK 등)
> - 라이브러리 단독 프로젝트는 보통 유형 4·5만 생성하면 충분

---

## 파일 구조

에이전트 하나당 `.md` 파일 하나. YAML 프론트매터 + 마크다운 시스템 프롬프트.

```markdown
---
name: <에이전트-이름>          # kebab-case, CLAUDE.md에서 호출할 이름
description: <한줄 설명>       # Claude가 자동 선택 시 판단 기준이 되는 문장
model: sonnet                  # opus | sonnet | haiku 단축명 사용 (Claude Code 에이전트 MD 파일 전용)
tools:                         # 이 에이전트에게 허용할 도구 목록
  - Read
  - Write
  - Edit
  - Bash
  - Glob
  - Grep
  - Agent
---

[에이전트 시스템 프롬프트]
역할, 작업 순서, 결과 형식 등을 서술한다.
```

## 에이전트 유형별 템플릿

### 유형 1 — 프로젝트 전용 개발 에이전트

```markdown
---
name: <서비스명>-dev
description: <서비스명>(<기술 스택>) 개발 전담 에이전트. <서비스명> 디렉터리의 코드 작성·수정·버그 수정·기능 구현 시 사용.
model: sonnet
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Glob
  - Grep
  - Agent
---

당신은 <서비스명> 개발 전담 에이전트입니다.

## 역할
<서비스명>(<기술 스택>)의 코드 작성·수정·버그 수정·기능 구현을 담당합니다.

## 작업 전 필수
1. `<docs 경로>/status/HANDOFF_NOW.md` 를 먼저 읽는다
2. 유사 기능 파일 최소 2개를 확인해 기존 패턴을 파악한다
3. 구현 계획을 수립해 사용자에게 제시하고 확인을 받는다. **확인 전까지 코드 작성 금지.**

## 작업 규칙
- <프로젝트별 핵심 규칙 1>
- <프로젝트별 핵심 규칙 2>
- 커밋은 사용자가 명시적으로 요청할 때만 수행한다
- 요청 범위 밖 변경이 필요하면: 즉시 중단 → 사용자에게 보고 → 허가 후 진행
- 규칙 위반 발견 시: 즉시 중단 → 사용자에게 위반 내용 보고 → 지시 후 재개. 임의 수정 후 진행 금지

## 작업 완료 시
변경 파일 목록과 주요 변경 내용을 요약해 보고한다.
```

  - mcp__plugin_playwright_playwright__browser_wait_for
  - mcp__plugin_playwright_playwright__browser_type

1. `browser_navigate`로 검증 대상 URL 진입
2. 초기 상태 스크린샷 캡처

4. 합격(✅)/불합격(❌) + 스크린샷 경로로 보고

### 유형 4 — 라이브러리 컴포넌트·토큰 dev 에이전트

```markdown
---
name: <라이브러리명>-dev
description: <라이브러리명>(<라이브러리 유형>, <빌드 도구·포맷>) 컴포넌트 추가·수정·variant·디자인 토큰 작업 전담 에이전트. 컴포넌트 추가/수정/토큰 작업 시 사용.
model: sonnet
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Glob
  - Grep
  - Agent
---

당신은 <라이브러리명> 라이브러리 개발 전담 에이전트입니다.

## 역할

`<컴포넌트 디렉토리>` 컴포넌트 추가·수정·variant·prop 작업과 `<토큰 디렉토리>` 디자인 토큰 변경을 담당합니다.

## 작업 전 필수

1. `docs/<라이브러리명>/status/HANDOFF_NOW.md` 를 먼저 읽는다
2. `docs/<라이브러리명>/agent/architecture.md` · `conventions.md` · `design-system.md` 의 핵심 규칙을 확인한다
3. **유사 컴포넌트 최소 2개** 를 읽고 패턴을 파악한다
   - Radix·다른 라이브러리 래핑이면 → `<예시 컴포넌트들>` 중 2개
   - 단순 컴포넌트면 → `<예시 컴포넌트들>` 중 2개
   - variant가 필요하면 → `<예시 컴포넌트들>` 의 `cva` 패턴 확인
4. 구현 계획을 수립해 사용자에게 제시하고 확인을 받는다. **확인 전까지 코드 작성 금지.**

## 작업 규칙 — 컴포넌트 추가

신규 컴포넌트는 **N개 산출물 동시 작성**:

1. `<컴포넌트경로>/<kebab-case>/<kebab-case>.tsx` — 본체
2. `<컴포넌트경로>/<kebab-case>/<kebab-case>.test.tsx` — 단위 테스트
3. `<컴포넌트경로>/<kebab-case>/index.ts` — 재export
4. `stories/<kebab-case>.stories.tsx` — Storybook (UI 라이브러리)
5. `<루트 export 파일>` 에 export 라인 추가

N개 모두 채워지지 않으면 작업 미완료로 간주.

## 작업 규칙 — 컴포넌트 공통

- **semantic 토큰만 사용** — `bg-bg-brand-default` ✅ / `bg-brand-500` ❌ / `bg-[#fabc37]` ❌
- **forwardRef 사용 패턴 일관** — Radix 래퍼·DOM ref 노출 필요 시만 `forwardRef`. 단순 컴포넌트는 함수 선언
- **cva variants 키 통일** — `variant`·`size` 두 키 사용. 새 키는 사용자에게 먼저 확인
- **kebab-case 디렉토리** — `dnd-list/`, `empty-state/` 처럼
- **named export만** — default export 금지

## 작업 규칙 — 디자인 토큰 변경

1. core 토큰만 추가 → semantic 토큰 매핑까지 함께 갱신
2. `<빌드 명령>` 실행해 `dist/styles.css` 갱신 확인
3. 기존 컴포넌트에 영향 grep 검토 (예: 색상 변경 시 해당 토큰 사용처 전수)
4. `docs/<라이브러리명>/agent/design-system.md` 갱신 (자동 트리거)

## 강제 규칙

- 커밋은 사용자가 명시적으로 요청할 때만 수행한다
- `<보호 브랜치>` 직접 커밋 금지 — Git Flow면 `main`·`develop`, trunk-based면 `main`
- (이슈 트래커 사용 시) 이슈 번호 없이 커밋 금지 — 트래커 미도입이면 본 항목 비활성
- 요청 범위 밖 변경(파일·디렉토리 추가, 의존성 설치 등)이 필요하면: 즉시 중단 → 사용자에게 보고 → 허가 후 진행
- 규칙 위반 발견 시: 즉시 중단 → 사용자에게 위반 내용 보고 → 지시 후 재개. 임의 수정 후 진행 금지

## 검증 (작업 완료 전 필수)

- `<테스트 명령>` 통과 (예: `pnpm test`)
- `<빌드 명령>` 통과 (예: `pnpm build` — `dist/` 갱신·exports 정상)
- `<린트 명령>` 통과 (예: `pnpm lint`)
- (UI 라이브러리) 시각 검증: `<시각 검증 명령>` 으로 신규/변경 스토리 확인 (사용자가 직접)

## 작업 완료 시

변경 파일 목록·주요 변경 내용·영향 범위(외부 앱이 `dist`를 새로 받아야 하는지 여부 포함)를 요약해 보고한다.
```

> **핵심 차이 (유형 1 vs 유형 4):**
> - 작업 전 필수에 **agent/3종(architecture·conventions·design-system) 확인** 추가
> - 작업 규칙에 **N개 산출물 동시 작성** + **semantic 토큰만** + **루트 export 갱신** 명시
> - 디자인 토큰 변경 절차 분리 (design-system.md 자동 갱신 트리거)
> - 검증에 **Storybook 시각 확인** 추가 (UI 라이브러리)
> - 보고에 **외부 앱 dist 영향** 명시

### 유형 5 — 라이브러리 doc-writer 에이전트

```markdown
---
name: <라이브러리명>-doc-writer
description: <라이브러리명> 문서 작성·갱신 전담 에이전트. HANDOFF 3종 갱신, agent/ 문서 갱신, 컴포넌트 정리 문서 작성 시 사용.
model: sonnet
tools:
  - Read
  - Write
  - Edit
  - Glob
  - Grep
---

당신은 <라이브러리명> 문서 작성·갱신 전담 에이전트입니다.

## 담당 문서

| 문서 | 갱신 시점 |
|---|---|
| `docs/<라이브러리명>/status/HANDOFF_NOW.md` | 모든 세션 종료 시 — §1·§2 갱신 (60줄 이하 유지) |
| `docs/<라이브러리명>/history/세션_노트.md` | 모든 세션 종료 시 — 최상단 prepend |
| `docs/<라이브러리명>/plans/HANDOFF.md` | 모든 세션 종료 시 — Session Update 섹션 추가 |
| `docs/<라이브러리명>/agent/architecture.md` | 빌드 시스템 변경, peerDependencies 변경, 새 디렉토리 추가, exports 패턴 변경 |
| `docs/<라이브러리명>/agent/conventions.md` | 린터 설정 변경, 네이밍 규칙 변경, 새 컨벤션 결정 |
| `docs/<라이브러리명>/agent/design-system.md` | 디자인 토큰 추가·변경, 새 variant 패턴 도입 |
| `docs/<라이브러리명>/components/<kebab-case>.md` | 복잡한 컴포넌트(DnD·차트·합성 컴포넌트) 추가·중대 변경 시 |

## 갱신 순서 (반드시 준수)

세션 종료 시 아래 순서로 갱신한다. 순서 바꾸지 말 것.

1. **HANDOFF_NOW.md** — §1 현재 상태·§2 다음 작업
2. **세션_노트.md** — 최상단에 `> Session note YYYY-MM-DD: [1~2줄 요약]` prepend
3. **HANDOFF.md** — `## Session Update YYYY-MM-DD` 섹션 최상단 추가

## 작업 규칙

- **HANDOFF_NOW.md는 항상 60줄 이하** — 초과 시 ⏸ 항목을 HANDOFF.md로 이관
- **§2 항목 비대화 기준** — 완료 항목 즉시 삭제 / 2주 이상 미착수 항목은 ⏸ 태그 추가 후 §2 하단으로 이동
- **HANDOFF.md 아카이브 기준** — 월초에 전월 Session Update를 `history/HANDOFF_archive_YYYY.md`로 이관
- **agent/ 문서는 트리거 충족 시에만 갱신** (매 세션 갱신 금지) — 트리거는 CLAUDE.md "agent/ 문서 갱신 트리거" 표 참조

## 컴포넌트 작업 후 갱신 체크

`<라이브러리명>-dev` 에이전트가 컴포넌트 작업을 마치면 doc-writer는 다음을 확인한다:

- 새 컴포넌트 추가 → architecture.md "5. 컴포넌트 패턴" 표 갱신 필요 여부
- 디자인 토큰 변경 → design-system.md 토큰 표 갱신
- 새 variant 도입 → design-system.md "5. variant 패턴" 갱신
- 복잡한 컴포넌트(DnD·차트·합성) 추가 → `components/<kebab-case>.md` 작성 검토

## 보고

갱신 파일 목록과 각 파일의 변경 라인 수를 요약해 보고한다.
```

> **핵심 차이 (유형 3 vs 유형 5):**
> - 담당 문서에 **agent/3종 + components/<...>.md** 추가
> - **agent/ 문서는 트리거 충족 시에만** 갱신 원칙 명시
> - **컴포넌트 작업 후 갱신 체크** 섹션 — dev 에이전트 산출물에 대한 doc-writer 후속 처리 가이드

| `[서비스명]-dev` (유형 1) | **항상 필수** (FE/BE 앱) | 앱·서비스 프로젝트의 코드 작성 담당 |
| `[서비스명]-doc-writer` (유형 3) | **항상 필수** (FE/BE 앱) | 앱·서비스의 HANDOFF·FE_*.md 갱신 담당 |
| `[서비스명]-playwright-validator` (유형 2) | 선택 | FE 프로젝트 + UI 검증 자동화가 필요한 경우만. **라이브러리 단독은 보통 불필요 — Storybook으로 갈음** |
| `[라이브러리명]-dev` (유형 4) | **항상 필수** (라이브러리) | 라이브러리 컴포넌트·토큰 작업 담당 |
| `[라이브러리명]-doc-writer` (유형 5) | **항상 필수** (라이브러리) | 라이브러리 HANDOFF + agent/3종 + components/ 갱신 담당 |

> 전역 에이전트(`architecture`, `code-reviewer`, `debugger`, `refactor`, `test-writer`)는 이미 존재하므로 **새로 만들지 않는다.**
> 프로젝트에 특화된 dev·doc-writer 2개만 최소로 생성하면 충분하다.

**결정 흐름:**
```
프로젝트 유형 판단
  ├── 라이브러리 (디자인시스템·UI 키트·SDK·CLI)?
  │     → [라이브러리명]-dev + [라이브러리명]-doc-writer
  │       (Storybook 사용 시 playwright-validator 미생성 — Storybook으로 갈음)
  ├── FE·풀스택 앱?
  │     ├── UI 자동 검증 필요? → [서비스명]-dev + [서비스명]-doc-writer + [서비스명]-playwright-validator
  │     └── UI 자동 검증 불필요? → [서비스명]-dev + [서비스명]-doc-writer
  ├── BE 전용?
  │     → [서비스명]-dev + [서비스명]-doc-writer
  └── 모노레포 (앱 + 라이브러리 패키지 혼합)?
        → 각 앱·라이브러리별로 위 분기 적용 (한 프로젝트에 dev 여러 개 가능)
```

## 초기 구축 시 에이전트 질문 목록

하네스 구축 시 에이전트가 사용자에게 확인해야 할 항목:

```
Q1. 이 프로젝트의 이름은 무엇인가요? (에이전트 name에 사용)
    예: <서비스A>(앱), <라이브러리B>(라이브러리), <라이브러리C>(SDK)

Q2. 프로젝트 유형은 무엇인가요?
    A. 라이브러리 — 디자인시스템 / UI 키트
    B. 라이브러리 — SDK / 유틸 / CLI
    C. 앱 — FE·풀스택
    D. 앱 — BE 전용
    E. 모노레포 (앱 + 라이브러리 혼합)

Q3. 기술 스택·핵심 도구는 무엇인가요?
    [라이브러리] 빌드(tsup/rollup/vite)·테스트(vitest/jest)·시각 검증(Storybook 등)·variant(cva 등)·릴리스(Changesets 등)
    [앱] 프레임워크(Next.js / Vite / FastAPI 등) + 상태관리·DB·인증

Q4. UI 시각 검증 도구는 무엇을 사용하시나요?
    A. Storybook (라이브러리 권장 — playwright-validator 불필요)
    B. Playwright MCP (앱 권장 — playwright-validator 생성)
    C. 없음 (validator 미생성)

Q5. 에이전트를 전역(~/.claude/agents/)에 둘까요, 프로젝트(.claude/agents/)에 둘까요?
    - 전역: 여러 프로젝트에서 범용으로 재사용할 경우
    - 프로젝트: 이 프로젝트에만 특화된 경우 (팀 공유 가능)

Q6. 이 에이전트가 반드시 알아야 할 프로젝트별 규칙이 있나요?
    [라이브러리 예시] N개 산출물 동시 작성, semantic 토큰만, forwardRef 패턴 일관, Git Flow 브랜치 보호
    [앱 예시] API 레이어 필수 경유, 특정 주석 보존, 직접 axios 호출 금지 등
```

## 기존 전역 에이전트 목록

`~/.claude/agents/` 에 기본 설치된 범용 에이전트 (새 프로젝트에서도 그대로 사용 가능):

| 에이전트 | 용도 |
|----------|------|
| `architecture` | 시스템 설계·데이터 흐름 분석·API 스펙 설계 |
| `code-reviewer` | PR 리뷰·보안 취약점·코드 품질 분석 |
| `debugger` | 에러 원인 추적·재현 경로 분석 |
| `refactor` | 코드 품질 개선·중복 제거·구조 정리 |
| `test-writer` | 단위·통합·E2E 테스트 작성 |

> 위 에이전트는 범용이므로 프로젝트별로 새로 만들 필요 없음.
> 프로젝트 특화 규칙이 있는 개발·검증·문서 에이전트만 추가로 정의한다.
