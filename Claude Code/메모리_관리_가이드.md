# Claude Code 메모리 관리 가이드

`~/.claude/projects/<프로젝트경로>/memory/` 에 자동 저장되는 영구 메모리 운영 방법.
세션이 바뀌어도 Claude가 기억해야 할 내용을 저장·관리한다.

---

## 메모리 저장 위치

```
~/.claude/projects/
└── <프로젝트경로-슬래시를-하이픈으로>/
    └── memory/
        ├── MEMORY.md          ← 메모리 인덱스 (항상 로드됨)
        ├── feedback_*.md      ← 작업 방식 피드백
        ├── user_*.md          ← 사용자 정보·선호도
        ├── project_*.md       ← 프로젝트 현황·결정사항
        └── reference_*.md     ← 외부 리소스 위치
```

**경로 형식 예시:**

| 프로젝트 절대경로 | 메모리 폴더 경로 |
|------------------|----------------|
| `/Users/<사용자명>/<프로젝트명>` | `~/.claude/projects/-Users-<사용자명>-<프로젝트명>/memory/` |
| `/Users/xxx/myapp` | `~/.claude/projects/-Users-xxx-myapp/memory/` |

> **자동 생성**: Claude Code가 프로젝트에서 첫 세션을 열면 `projects/` 하위 폴더가 자동 생성된다. 수동 mkdir 불필요.
> **MEMORY.md 주의**: 200줄 이하로 유지. 초과 시 오래된 항목을 개별 파일로 이관하고 MEMORY.md에는 링크만 남긴다.

---

## 메모리 유형

| 유형 | 파일 접두사 | 저장 내용 | 예시 |
|------|------------|-----------|------|
| `user` | `user_*.md` | 사용자 역할·선호도·배경 지식 | 시니어 FE 개발자, 한국어 선호 |
| `feedback` | `feedback_*.md` | 작업 방식 교정·확인된 접근법 | 커밋 자동 수행 금지, worktree 자제 |
| `project` | `project_*.md` | 진행 중 작업·결정사항·데드라인 | API 마이그레이션 2026-05-01 완료 목표 |
| `reference` | `reference_*.md` | 외부 리소스 위치 | Linear 프로젝트 ID, Figma 파일 URL |

## 저장해야 할 것 vs 저장하지 말 것

### 저장 O

| 상황 | 저장 내용 |
|------|-----------|
| 사용자가 작업 방식을 교정했을 때 | `feedback_*.md` — 규칙 + 이유 + 적용 시점 |
| 사용자가 비명시적으로 확인한 접근법 | `feedback_*.md` — 검증된 판단 기록 |
| 프로젝트 특수 결정사항 | `project_*.md` — 결정 + 이유 + 적용 방법 |
| 반복 참조하는 외부 리소스 | `reference_*.md` — URL + 용도 |
| 사용자가 "기억해줘"라고 한 것 | 해당 유형으로 즉시 저장 |

### 저장 X (코드/Git으로 관리)

| 항목 | 대신 사용 |
|------|-----------|
| 코드 패턴·아키텍처·파일 경로 | 코드 자체 / 주석 |
| 최근 커밋·변경 이력 | `git log` / `git blame` |
| 세션 내 임시 상태 | TodoWrite |
| CLAUDE.md에 이미 있는 규칙 | 중복 저장 금지 |
| 현재 진행 중인 태스크 | TodoWrite |

## feedback 메모리 작성 형식

```markdown
---
name: [규칙명]
description: [한줄 설명 — 미래 세션에서 관련성 판단 기준]
type: feedback
---

[규칙 본문]

**Why:** [사용자가 교정한 이유]
**How to apply:** [언제·어디서 이 규칙을 적용할지]
```

**예시:**
```markdown
---
name: 커밋 자동 수행 금지
description: 사용자 명시적 요청 없이 커밋 수행 금지
type: feedback
---

코드 작업 완료 후 자동으로 커밋하지 않는다. 사용자가 "커밋해줘"라고 명시적으로 요청할 때만 수행한다.

**Why:** 검수 전 상태로 커밋되는 사고가 발생했음.
**How to apply:** 코드 수정 완료 후 항상 결과 요약만 제공하고 커밋은 대기.
```

## project 메모리 작성 형식

```markdown
---
name: [프로젝트 결정사항명]
description: [한줄 설명]
type: project
---

[결정 내용]

**Why:** [결정 배경·제약 조건]
**How to apply:** [이 결정이 작업에 미치는 영향]
```

## 초기 구축 시 에이전트 질문 목록

```
Q1. 이 프로젝트에서 Claude가 반드시 기억해야 할 작업 규칙이 있나요?
    예: 커밋 자동 금지, 특정 패턴 사용, 특정 에이전트 우선 등

Q2. 자주 참조하는 외부 리소스가 있나요?
    예: Linear 프로젝트 URL, Figma 파일 URL, Slack 채널명

Q3. Claude가 알아야 할 팀/사용자 특성이 있나요?
    예: 백엔드 경험 있지만 React 신규, 한국어 응답 선호, 요약 없는 응답 선호

Q4. 진행 중인 결정사항 중 배경 설명이 필요한 것이 있나요?
    예: 특정 기술 선택 이유, 리팩토링 방향, 아키텍처 결정
```

## 메모리 vs HANDOFF 구분 기준

| 정보 유형 | 저장 위치 |
|-----------|-----------|
| 세션 간 이월되는 **작업 규칙·교정** | `memory/feedback_*.md` |
| 세션 간 이월되는 **프로젝트 결정** | `memory/project_*.md` |
| 현재 세션의 **작업 상태·다음 할 일** | `HANDOFF_NOW.md` §2 |
| 과거 세션의 **변경 이력** | `HANDOFF.md` |
| 세션 요약 **1~2줄** | `세션_노트.md` |

> 메모리는 **왜·어떻게** 를 기억하는 곳. HANDOFF는 **무엇을 했고 무엇을 해야 하는지** 를 기억하는 곳.
