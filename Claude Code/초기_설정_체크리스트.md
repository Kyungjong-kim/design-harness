# 신규 프로젝트 하네스 초기 설정 체크리스트

새 프로젝트에서 Claude Code 하네스를 처음 구축할 때 따르는 순서표.
각 단계는 독립적이지 않으며 **순서대로** 진행해야 한다.

> **이 문서의 역할**
>
> `/project-init` 스킬의 **상세 구현 기준**이다.
> `/project-init`을 실행하면 스킬이 이 체크리스트의 Phase 1~10을 순서대로 수행한다.
>
> 직접 읽어야 하는 경우:
> - 특정 Phase의 세부 기준을 확인할 때
> - 에이전트가 구축한 결과를 검토할 때
> - 수동으로 일부 Phase만 재실행할 때

> **에이전트 사용 방법**: "이 프로젝트에 하네스를 구축해줘. `하네스/Claude Code/초기_설정_체크리스트.md` 참고해서." 라고 하면 에이전트가 Phase 0부터 순서대로 수행한다.

---

## Phase 0 — 스킬 설치 확인 (신규 환경)

`/project-init`을 처음 실행하거나 새 기기에서 하네스를 구축할 때만 수행한다.
**하네스 스킬이 이미 설치된 환경이면 Phase 1로 바로 진행한다.**

- [ ] **0-A. 스킬 설치 여부 확인**
  ```bash
  HARNESS_SKILLS=(project-init project-fix project-issue project-pr session-close document-review)
  for skill in "${HARNESS_SKILLS[@]}"; do
    printf "%-25s" "$skill:"
    if [ -d "$HOME/.claude/skills/$skill" ]; then echo "전역 ✓"
    elif [ -d ".claude/skills/$skill" ]; then echo "로컬 ✓"
    else echo "✗ 미설치"
    fi
  done
  ```

- [ ] **0-B. 미설치 스킬이 있을 경우 — 설치 방법 선택**

  | 방법 | 명령 | 특징 |
  |------|------|------|
  | 전역 설치 (권장) | `bash 하네스/install.sh` | 이 기기 모든 프로젝트에서 사용. 훅·model 설정 포함 |
  | 로컬 설치 | `bash 하네스/install.sh --local` | `.claude/skills/` 생성. git 추적 시 팀 공유 가능 |

  > **로컬 설치 선택 시 git 추적 권장 파일:**
  > - `.claude/skills/` — 스킬 본문 (팀 공유 핵심)
  > - `.claude/settings.json` — harness 경로 additionalDirectories 포함
  > `.gitignore`에서 `.claude/` 제외 여부 확인.

Phase 0 완료 후 Phase 1로 진행한다.

## Phase 1 — 프로젝트 분석 (사용자 개입 없음)

에이전트가 코드를 읽고 아래 항목을 파악한다. **이 단계에서는 파일을 수정하지 않는다.**

- [ ] **1-A. 프로젝트 기술 스택 파악**
  - `package.json` / `pyproject.toml` / `requirements.txt` 확인
  - 프레임워크, 주요 라이브러리, 빌드 도구 파악

- [ ] **1-B. 소스 디렉토리 구조 파악**
  - `find . -maxdepth 4 -type d` 로 전체 폴더 구조 확인
  - **폴더 수 100개 초과 시**: 결과가 컨텍스트를 압도할 수 있으므로 `Explore` 서브에이전트로 위임 권장
    ```
    Explore: "프로젝트 유형(단일앱/모노레포), 주요 소스 경로, 도메인 목록을 확인해줘"
    ```
  - 아래 판단 기준으로 프로젝트 유형 결정:

  | 판단 기준 | 결론 |
  |-----------|------|
  | `package.json`에 `workspaces` 키 있음 | 모노레포 |
  | `apps/` 또는 `packages/` 하위에 독립 `package.json` 복수 존재 | 모노레포 |
  | `src/` 하나만 존재 | 단일 앱 |
  | 위 어느 것도 해당 없음 | Q&A로 확인 |

  - 도메인 개수 파악: `pages/`, `views/`, `features/` 하위 1단계 폴더 목록화

  **다중 패키지 모노레포 에이전트 분리 기준:**

  | 조건 | 에이전트 구성 |
  |------|-------------|
  | 패키지 수 ≤ 2개 | 패키지마다 별도 에이전트 (`web-dev.md`, `admin-dev.md`) |
  | 패키지 수 ≥ 3개 | 핵심 2~3개만 별도 에이전트. 나머지는 단일 공통 에이전트 |
  | 패키지 간 기술스택 동일 | 단일 에이전트 + 패키지명 분기 규칙 기재 가능 |
  | 패키지 간 기술스택 상이 | 패키지마다 별도 에이전트 필수 |

  > 에이전트 파일명: `<패키지명>-dev.md` (예: `web-dev.md`, `api-dev.md`)

- [ ] **1-C. 기존 문서·설정 확인**
  - `README.md`, `CONTRIBUTING.md` 검토
  - 기존 `CLAUDE.md` / `.claude/` 여부 확인 (있으면 내용 파악 후 충돌 지점 메모)
  - `git log --oneline -20` → 커밋 접두사 패턴 파악 (`[FE]`, `feat:` 등)

- [ ] **1-D. 팀 프로세스 파악**
  - 브랜치 전략: `git branch -a` + git log 패턴으로 추론
  - 이슈 트래커: `README.md` 또는 `.github/` 폴더에서 확인
  - 커밋 컨벤션: 최근 커밋 10개에서 패턴 추출

**Phase 1 완료 조건**: 아래 항목을 채울 수 있어야 다음으로 진행한다.
```
- 프로젝트 유형: [단일 앱 / 모노레포]
- FE/BE 여부: [FE / BE / 풀스택]
  판단 기준: React·Vue·Next.js 등 UI 프레임워크 존재 시 FE
             FastAPI·Django·Express 등 서버 프레임워크만 있으면 BE
- 주요 소스 경로: [경로 목록]
- 도메인 목록: [pages/ 하위 폴더명들]
- 커밋 접두사 패턴: [추론된 패턴 or 불명확]
- 이슈 트래커: [확인됨 / 불명확]
```

## Phase 2 — Q&A 수집 (사용자에게 질문)

Phase 1에서 확인하지 못한 항목을 사용자에게 질문한다.
**질문은 한 번에 모아서 한다. 개별 단계마다 묻지 않는다.**

```
[필수 질문]
Q1. 이 서비스 이름을 뭐로 할까요?
    → docs 폴더명, 에이전트 이름, 커스텀 에이전트 파일명에 사용됩니다.
    예: web-app, portal, dashboard
    → 모노레포인 경우: 구축할 패키지명 입력 (예: web, admin). 패키지가 여러 개면 하나씩 순차 진행.

Q2. 주로 작업할 영역은? (FE / BE / 풀스택)
    → Phase 5 에이전트 컨텍스트 문서·Phase 6 커스텀 에이전트 정의 항목에 영향.

Q3. CLAUDE.md를 프로젝트 루트에 둘까요, 아니면 .gitignore 처리할까요?
    A) 팀 공유 (추적) — git 관리, 팀 전체 적용
    B) 개인 설정 (gitignore) — 로컬에만 적용

Q4. 커스텀 에이전트를 어디에 둘까요?
    A) 전역 (~/.claude/agents/) — 여러 프로젝트 재사용
    B) 프로젝트 (.claude/agents/) — 이 프로젝트 전용, 팀 공유 가능

Q5. 브랜치 전략은?
    → Phase 1에서 git log 분석 결과를 먼저 제시하고 맞는지 확인한다.
      분석으로 확인이 안 됐거나 신입인 경우 팀장에게 먼저 확인 후 답변.

    일반적인 패턴:
    - develop 기준: develop에서 분기 → task/<이슈번호> → develop으로 PR
    - main 기준:    main에서 분기 → feature/<이슈번호> → main으로 PR
    - 트렁크 기반:  main에 직접 커밋 (소규모 팀)

    확인 항목: base 브랜치명 / feature 브랜치 네이밍 규칙 / PR 대상 브랜치

Q6. Claude Code가 프로젝트 외부에서 접근해야 하는 디렉터리가 있나요?
    예: Obsidian Vault 경로, 공유 컨벤션 저장소 등
    실제 경로를 알려주세요. (예: /Users/홍길동/Documents/Dev-Vault)
    없으면 "없음"이라고 해주세요.
    → install.sh를 사용했다면 하네스 폴더는 이미 자동 등록됨. 위 경우에 해당하지 않으면 "없음".

[확인이 필요한 경우만 추가 질문]
Q7. (커밋 패턴 불명확 시) 커밋 접두사 규칙이 있나요?
Q8. (이슈 트래커 불명확 시) 이슈 트래커는 무엇인가요?
Q9. (모노레포인 경우) docs 위치를 루트 통합형과 패키지별 중 어느 걸 선호하시나요?
    → 선택 기준: `하네스/Claude Code/docs_디렉토리_구조.md` 참고
Q10. Playwright MCP UI 검증 에이전트가 필요한가요?
```

## Phase 3 — docs 폴더 구조 생성

Q&A에서 확인한 서비스명과 docs 위치로 폴더를 생성한다.
참조: `하네스/Claude Code/docs_디렉토리_구조.md`

- [ ] **3-0. 기존 docs/ 구조 감지 (먼저 실행)**

  ```bash
  ls docs/ 2>/dev/null && echo "기존 docs/ 존재" || echo "신규 생성 필요"
  find docs -maxdepth 3 -name "HANDOFF_NOW.md" 2>/dev/null
  ```

  아래 기준으로 경로를 결정한 뒤 3-A를 진행한다:

  | 감지 결과 | 처리 방법 |
  |---|---|
  | `docs/<서비스명>/status/` 구조 이미 존재 | `BASE=docs/<서비스명>` 설정 후 3-B로 이동 (3-A 건너뜀) |
  | `docs/status/` 구조 존재 (서비스명 없이) | 사용자에게 확인: A) 기존 경로 유지 → `BASE=docs` / B) 신규 구조 통일 → 기존 파일 이동 후 `BASE=docs/<서비스명>` |
  | docs/ 없음 또는 HANDOFF_NOW.md 없음 | 아래 3-A 명령으로 신규 생성 |

- [ ] **3-A. 필수 디렉토리 생성**
  ```bash
  SERVICE=<Q1에서 확인한 서비스명>

  # 단일 앱: 프로젝트 루트 기준
  mkdir -p docs/$SERVICE/{status,plans,history,agent,pages,git-workflow}

  # 모노레포: Q9 답변에 따라
  # 루트 통합형: mkdir -p docs/$SERVICE/{status,plans,history,agent,pages,git-workflow}
  # 패키지별:   mkdir -p <패키지경로>/docs/{status,plans,history,agent,pages,git-workflow}
  ```

- [ ] **3-B. 핵심 문서 초기화 (빈 파일만 생성)**
  ```bash
  BASE=docs/$SERVICE   # 또는 <패키지경로>/docs

  touch $BASE/status/HANDOFF_NOW.md
  touch $BASE/plans/HANDOFF.md
  touch $BASE/history/세션_노트.md
  touch $BASE/agent/README.md
  touch $BASE/agent/architecture.md
  touch $BASE/agent/conventions.md
  touch $BASE/git-workflow/branch-commit.md
  ```
  - `status/HANDOFF_NOW.md` — 내용은 Phase 9에서 채움 (`하네스/Claude Code/템플릿/HANDOFF_NOW_템플릿.md`)
  - `plans/HANDOFF.md` — 내용은 Phase 9에서 채움 (`하네스/Claude Code/템플릿/HANDOFF_템플릿.md`)
  - `history/세션_노트.md` — **`하네스/Claude Code/템플릿/세션_노트_템플릿.md`를 Read로 읽어 초기 1줄 복사해 넣기**
    (예: `> Session note YYYY-MM-DD: 하네스 초기 구축 완료 — CLAUDE.md·HANDOFF_NOW·agent/ 문서 작성, 훅 스크립트 경로 설정.`)
  - `agent/*.md` — 내용은 Phase 5에서 채움 (`하네스/Claude Code/템플릿/agent_문서_템플릿.md`)
  - `git-workflow/branch-commit.md` — 내용은 Phase 9-B에서 채움 (`하네스/Claude Code/템플릿/git_워크플로우_템플릿.md`)

## Phase 4 — CLAUDE.md 작성

Phase 1 분석 결과 + Phase 2 Q&A 답변으로 빈칸을 채운다.
**아래 파일을 Read로 먼저 읽어 템플릿을 확인한 뒤 작성한다:**
`하네스/Claude Code/템플릿/CLAUDE.md_템플릿.md`

- [ ] **4-0. 기존 CLAUDE.md 처리 (먼저 확인)**

  ```bash
  [ -f CLAUDE.md ] && echo "기존 CLAUDE.md 존재:" && head -5 CLAUDE.md || echo "신규 작성"
  ```

  - 없으면 → 4-A로 바로 진행
  - 있으면 → 사용자에게 처리 방법 확인:
    - **병합** (권장): 기존 팀 규칙을 유지하면서 하네스 필수 섹션
      (STEP 0 작업 시작 절차 · STEP 3 세션 종료) 추가
    - **대체**: 기존 내용을 `CLAUDE.md.bak`으로 백업 후 템플릿으로 새로 작성

- [ ] **4-A. 프로젝트 루트 `CLAUDE.md` 작성**
  - 저장 위치: `<project-root>/CLAUDE.md` (프로젝트 루트 최상단)
  - Phase 1-B에서 확인한 도메인 목록 → 0-A 작업 영역 테이블 작성
  - Phase 3에서 생성한 docs 경로 → 0-B 진입 문서 경로·STEP 3 갱신 순서에 반영
  - Phase 1-D에서 파악한 커밋 패턴 → 브랜치·커밋 규칙 섹션 반영

- [ ] **4-B. .gitignore 처리 (Q3 답변 기반)**
  - 팀 공유 → `.gitignore`에 추가하지 않음
  - 개인 설정 → `.gitignore`에 `CLAUDE.md` 추가

- [ ] **4-C. 스킬 목록 기재**
  - 실제 설치된 스킬 확인 후 CLAUDE.md `## 스킬` 섹션에 기재:
    ```bash
    ls ~/.claude/skills/ .claude/skills/ 2>/dev/null | sort -u
    ```
  - 설치 안 된 스킬은 행 삭제. 추가 설치된 스킬은 행 추가.
  - 기본 하네스 스킬: `session-close` · `project-fix` · `project-pr` · `project-issue` · `document-review`
  - 개인 스킬(--personal): `daily-note` · `standup` · `meeting-note` · `til` · `weekly-retro` · `weekly-meeting-update`

- [ ] **4-D. 서브패키지 CLAUDE.md 작성 (모노레포인 경우만)**
  - 루트 CLAUDE.md보다 더 제한적인 규칙이 필요한 패키지만 작성
  - 충돌 시 **더 제한적인 규칙**이 우선 (예: 루트는 "api/ 경유", 서브는 "useQuery 필수")

## Phase 5 — 에이전트 컨텍스트 문서 작성

Phase 1 분석 결과로 docs/agent/ 문서를 채운다.
**아래 파일을 Read로 먼저 읽어 템플릿을 확인한 뒤 작성한다:**
`하네스/Claude Code/템플릿/agent_문서_템플릿.md`

- [ ] **5-A. agent/README.md**
  - 이 프로젝트의 에이전트 목록과 각 파일의 역할을 한줄로 요약 작성
  - Phase 6에서 에이전트를 추가 정의한 뒤 목록 업데이트
  ```markdown
  # 에이전트 컨텍스트 문서
  - architecture.md — 앱 구조·인증·상태관리 패턴
  - conventions.md — 린터·커밋·네이밍 컨벤션
  - [서비스명]-dev.md — 코드 작성 전담 에이전트 정의
  - [서비스명]-doc-writer.md — 문서 갱신 전담 에이전트 정의
  ```

- [ ] **5-B. agent/architecture.md**

  - 이미 있으면 → **7-C 소스 경로 패턴 설정**으로 진행

- [ ] **7-C. hooks-config.sh 자동 생성**

  Phase 1-B에서 파악한 소스 경로로 `.claude/hooks-config.sh`를 생성한다.
  훅 스크립트가 이 파일을 자동 소싱하므로 개별 수동 설정 불필요.

  `/project-init` 실행 시 에이전트가 자동으로 처리하는 단계다.
  수동으로 실행할 경우 아래 절차를 따른다.

  **변수 결정 기준:**

  | 변수 | 결정 방법 |
  |---|---|
  | `FE_PATTERN` | Phase 1-B 소스 경로 + `(pages\|components\|hooks\|api)/([^/]+)/` |
  | `FE_DOC_BASE` | Phase 3 docs 경로 — `docs/<서비스명>/pages` |
  | `FE_PROJECT` | Q1 서비스명 |
  | `FE_COMMIT_PATTERN` | `^(<소스경로>)/.*\.(js\|ts\|tsx\|jsx)$` |
  | `FE_DOMAIN_EXTRACT` | `s\|<소스경로>/pages/([^/]+)/.*\|\1\|` |
  | `FE_DOC_DIR_BASE` | `docs/<서비스명>/pages` |

  BE·풀스택이면 `BE_*` 변수도 동일 방식으로 추가.

  생성 후 git 추적 필수:
  ```bash
  git add .claude/hooks-config.sh
  git commit -m "docs: [공통] 팀 훅 경로 설정 파일 추가"
  ```

  팀원은 `git pull` 후 자동 적용된다. 상세: `하네스/운영_가이드/팀_훅_설정_공유.md`
